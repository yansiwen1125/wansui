# 万岁 V1.0 技术方案

## 1. 评审结论

PRD 和高保真 UI 已具备开发条件。V1 推荐采用：

- 前端：React、TypeScript、Vite。
- PWA：`vite-plugin-pwa`，应用外壳离线缓存。
- 云端：Supabase Postgres 与 Supabase JavaScript SDK。
- 本地缓存：IndexedDB。
- 单元测试：Vitest。
- 页面自动化测试：Playwright。
- 发布：GitHub Pages，使用 Hash Router 避免刷新后 404。

V1 不建设自有服务器，不启用 Supabase Auth，不实现离线写入队列。

## 2. 已确认技术结论

以下问题已与产品经理逐项确认，开发和测试均以本表结论为准。

| 编号 | 岗位 | 问题 | 确认结论 |
| --- | --- | --- | --- |
| T1 | 前端 | PRD 写“保存成功后再更新”，任务卡片又写“点击后立即更新” | **已确认：**采用保存中状态；云端成功后更新界面，失败保持原状态 |
| T2 | 前端 | PRD 说日期箭头无按钮底框，但高保真稿存在浅灰按钮底色 | **已确认：**使用高保真稿中的浅灰色圆角按钮背景，不显示描边 |
| T3 | 前端 | 任务卡片左侧在 PRD 中叫“小方框”，高保真稿实际为竖向色块 | **已确认：**使用高保真稿中的任务颜色竖向圆角色块 |
| T4 | 前端 | 每个任务的三个月记录是独立滑动，还是全部任务同步月份 | **已确认：**每个任务独立维护三个月窗口；滑动时不影响其他任务 |
| T5 | 后端 | 取消打卡是删除记录，还是保存 `completed=false` | **已确认：**保留记录并写入 `completed=false`，支持最后成功写入生效 |
| T6 | 后端 | `effectiveStartDate` 按哪个时区生成 | **已确认：**不再动态计算，固定为 `2026-06-10` |
| T7 | 产品/测试 | 百分比如何取整 | **已确认：**页面四舍五入显示整数，业务判断使用未取整原值 |
| T8 | 产品/测试 | 今天未打卡且昨天也未打卡时，当前连续是多少 | **已确认：**为 `0`，不寻找更早的连续记录 |
| T9 | 测试 | 月历日期是否可点击进入首页对应日期 | **已确认：**不跳转、不选中、不修改状态，保持纯查看 |
| T10 | 后端 | 无 Auth 情况下是否接受匿名接口可被绕过 | **已确认：**V1 接受风险，不增加密码、验证码或 Supabase Auth |

## 3. 前端方案

### 3.1 页面与路由

使用 Hash Router：

- `#/login`：登录页。
- `#/home`：首页。
- `#/records/month`：月历。
- `#/records/tasks`：任务记录。

启动时先检查本地登录标记。没有标记进入登录页；存在标记进入首页并加载缓存与云端数据。

### 3.2 目录建议

```text
src/
├── app/
├── components/
├── pages/
├── features/checkins/
├── features/calendar/
├── features/statistics/
├── services/supabase/
├── storage/
├── domain/
├── styles/
└── tests/
```

### 3.3 单一数据源

前端内存中维护：

- 固定任务配置。
- 单用户配置。
- `taskId + date` 索引后的打卡记录。
- 当前选中日期。
- 当前月历月份。
- 每个任务各自的当前月份窗口。
- 网络与保存状态。

首页、月历和任务统计只读取同一份打卡数据，不能各自维护副本。

### 3.4 日期模块

所有日期使用设备本地日历日期字符串 `YYYY-MM-DD`，不使用 `toISOString()` 截取日期。

日期模块统一提供：

- 获取本地今天。
- 增减自然日和自然月。
- 获取周日至周六自然周。
- 生成月历 7 列网格。
- 判断今天、未来和固定生效日期 `2026-06-10` 边界。
- 生成三个月圆点矩阵。
- 处理闰年、跨月和跨年。

### 3.5 保存流程

1. 用户点击勾选。
2. 检查网络、日期范围和是否已有保存请求。
3. 对该按钮显示保存中状态并禁止重复点击。
4. 向 Supabase 执行 upsert。
5. 成功后更新内存数据和 IndexedDB。
6. 统一重新计算周摘要、月历和统计。
7. 失败则恢复按钮并显示错误提示，原状态不变。

同一个 `taskId + date` 同时只允许一个前端请求。

### 3.6 本地缓存与 PWA

- `localStorage`：仅保存登录标记和缓存版本号。
- IndexedDB：保存单用户配置和全部已读取打卡记录。
- Service Worker：缓存 HTML、JS、CSS、图标和字体等应用外壳。
- 启动时先显示缓存，再请求云端完整数据；云端成功后覆盖缓存。
- 离线时只读，不记录待同步操作。
- 新版本 Service Worker 下载完成后提示用户刷新，不强制中断当前操作。

数据量很小，V1 可缓存从 `effectiveStartDate` 至今的全部记录。

### 3.7 UI 实现原则

- 以高保真 SVG 为视觉基准，以 UI-DESIGN.md 为尺寸和状态基准。
- 页面使用移动端优先的响应式布局，`390px` 仅是设计基准，不作为固定页面宽度。
- 应用主体使用 `width: 100%`，宽屏设置 `max-width: 480px` 并水平居中。
- 页面左右内边距使用响应式值，例如 `clamp(16px, 6vw, 24px)`。
- 周摘要和月历使用 7 列等分网格，任务卡片使用可用宽度的 `100%`。
- 使用 CSS 变量维护色板、间距和圆角。
- 小方块和圆点使用固定宽高与 `aspect-ratio: 1`。
- iPhone 底部使用 `env(safe-area-inset-bottom)`。
- 月历与周摘要使用 CSS Grid。
- 每个任务的月份滑动独立实现横向吸附，一次切换一个自然月。
- 手势先判断主方向，避免横向月份和纵向页面滚动互相抢占。

## 4. 后端方案

### 4.1 表结构

#### `app_config`

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `id` | `smallint` | 固定为 `1`，主键 |
| `username` | `text` | 固定为 `ysw`，唯一 |
| `first_initialized_at` | `timestamptz` | 数据库生成 |
| `effective_start_date` | `date` | 固定为 `2026-06-10` |
| `schema_version` | `integer` | 默认 `1` |
| `last_known_time_zone` | `text` | 可空 |

#### `checkins`

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `task_id` | `text` | 仅允许 4 个固定 ID |
| `date` | `date` | 本地日历日期 |
| `completed` | `boolean` | 默认 `true` |
| `updated_at` | `timestamptz` | 数据库自动生成 |

主键为 `(task_id, date)`。

固定任务 ID：

- `preset_bowel`
- `preset_sleep`
- `preset_exercise`
- `preset_healthy_diet`

### 4.2 写入方式

完成和取消都使用同一条 upsert：

```sql
insert into checkins (task_id, date, completed)
values (:task_id, :date, :completed)
on conflict (task_id, date)
do update set
  completed = excluded.completed,
  updated_at = now();
```

客户端不能提交 `updated_at`。数据库以最后成功提交的事务为准。

### 4.3 初始化

首次登录调用初始化 RPC，参数为固定用户名和 IANA 时区，例如 `Asia/Shanghai`。

RPC 使用单事务和唯一键：

- 配置已存在：直接返回原配置。
- 配置不存在：创建一次，并将 `effective_start_date` 固定写为 `2026-06-10`。
- 后续设备不得修改 `first_initialized_at` 和 `effective_start_date`。

### 4.4 权限与已知风险

网页只能使用 Supabase publishable key，绝不包含 service role key。

由于 V1 不启用 Auth，数据库无法确认调用者真的是 Sally。可以通过约束、RPC 和最小表权限限制可写字段，但无法阻止知道项目接口的人读写这套数据。该风险只能通过启用真正身份验证解决。

### 4.5 查询

V1 数据量很小，登录后一次读取：

- 单用户配置。
- `effective_start_date` 至今的所有 checkins。

统计默认在前端纯函数中计算，避免页面与数据库各自实现一套口径。

## 5. 统计口径

所有统计均忽略未来日期和 `effectiveStartDate` 之前日期。

- 绿灯天数：当日完成数 ÷ 4 大于等于 50% 的有效日期数。
- 总完成率：本月完成记录数 ÷（4 × 本月有效日期数）。
- 总打卡：本月所有 `completed=true` 的记录数。
- 本月打卡率：单任务本月完成天数 ÷ 本月有效日期数。
- 本月次数：单任务本月 `completed=true` 的天数。
- 当前连续：今天完成则从今天向前；今天未完成则先检查昨天，昨天未完成时直接为 `0`。
- 最长连续：从生效日期到今天扫描全部完成日期得到的最大连续段。

百分比展示使用 `Math.round(value * 100)`；是否达到 50% 使用原始比例判断。

## 6. 测试方案

### 6.1 自动化分层

- 单元测试：日期、月历矩阵、统计、编辑范围。
- 组件测试：任务卡片、田字格、九宫格、按钮状态。
- 集成测试：Supabase 请求成功、失败、重复写入和缓存更新。
- E2E：登录、打卡、取消、切换日期、查看月历、查看任务记录。
- 真机测试：iPhone Safari 与添加到主屏幕后运行。

### 6.2 固定测试数据

建立测试数据生成器，至少覆盖：

- 2024 年闰年二月。
- 1 日为周日的月份。
- 1 日为周六且需要 6 行的月份。
- 跨年三个月窗口。
- 连续、断签、取消历史打卡。
- 50% 完成率边界。
- 生效日期位于月中。

测试数据入口只存在于测试构建，不进入正式版本。

### 6.3 发布门槛

- 日期和统计单元测试全部通过。
- 核心 E2E 全部通过。
- Supabase 重复记录、丢失记录和错误日期问题为零。
- 当前主流 iPhone Safari 无阻塞问题。
- `320px–480px` 宽度范围内布局可以连续自适应，无内容截断或异常留白。
- 离线只读、保存失败和重新联网恢复正常。
- PWA 安装、启动和版本更新正常。

## 7. 开发任务拆分

### 阶段一：工程与数据

1. 创建 React + TypeScript + Vite 工程。
2. 配置 GitHub Pages、PWA 和环境变量。
3. 创建 Supabase 表、约束、触发器和初始化 RPC。
4. 完成固定任务、日期和统计纯函数。

### 阶段二：核心页面

1. 登录页与登录状态。
2. 首页日期切换、周摘要和打卡。
3. 月历页。
4. 任务记录页与月份滑动。

### 阶段三：缓存与异常

1. IndexedDB 缓存。
2. 离线只读。
3. 加载、保存中和失败提示。
4. Service Worker 更新提示。

### 阶段四：测试与发布

1. 日期和统计自动化测试。
2. 核心流程 E2E。
3. iPhone Safari 真机验收。
4. GitHub Pages 正式部署。

## 8. 推荐开发顺序

先完成“登录 → 首页打卡 → 云端保存 → 重新打开仍存在”的最小闭环，再开发月历和任务统计。日期与统计测试通过后才进入正式 UI 验收，避免在错误数据基础上反复调整页面。
