# 万岁 V1.0 前端需求对齐

## 角色

**前端工程师：阿序**

负责将 PRD 和设计稿实现为可安装网页 App，包括页面、交互、日期计算、PWA、本地缓存、云端读写和统计展示。

## 已对齐需求

- 两个一级页面：首页、记录页。
- 记录页包含「月历」和「任务」两个 Tab。
- 打卡支持 `2026-06-10` 起（含当天）至今天之间的日期，不允许进入未来或更早日期。
- 所有页面共用同一份任务顺序和打卡数据。
- 数据修改后，首页周摘要、月历、总览和任务统计必须同步更新。
- V1 使用固定用户名 `ysw` 进入，并将单用户数据同步到 Supabase。

## 技术与安装结论

- 使用移动端优先的 PWA。
- 页面宽度采用响应式布局，不固定为设计稿的 `390px`。
- 应用主体宽度为 `100%`，宽屏使用最大宽度限制并水平居中。
- 代码托管在 GitHub，通过 GitHub Pages 发布。
- iPhone Safari 可将网页添加到主屏幕。
- 使用 Supabase JavaScript 客户端完成单用户数据读写和同步。
- 使用 Service Worker 缓存应用外壳，使用 IndexedDB 保存最近一次云端数据快照。
- 使用浏览器本地存储保存 `ysw` 已登录标记。

## 实现建议

- 将 4 个预制任务放在统一静态配置中，不从云端创建或修改；所有页面只读取该配置。
- 日期统一使用设备本地日历日期 `YYYY-MM-DD`，不要直接用 UTC 日期作为展示日期。
- 可编辑日期范围为固定生效日期 `2026-06-10` 到今天，两个边界均包含。
- 自然周计算明确以周日为第一天。
- 打卡状态采用单一数据源，勾选后统一重新计算各类摘要。
- 月历组件和任务圆点月历分别封装，避免复制日期算法。
- 横向月份滑动与纵向任务滚动要设置清晰的手势方向判断。
- 月历最多只生成当前可见月份附近的数据，避免无边界渲染。
- 本地存储必须带数据版本号，后续版本可以迁移。
- 前端只能使用 Supabase publishable key，不得包含 service role key。
- 用户名判断只用于控制界面入口，不得在代码注释或产品文案中宣称它具备账号安全性。
- 云端只维护一套单用户数据；知道网页地址的人仍可能绕过前端入口直接请求公开接口。
- 离线时只允许读取缓存，不建立待同步操作队列。
- 打卡请求成功后再更新界面；失败时保持原状态。
- `updatedAt` 使用数据库生成的时间，不使用手机时间。

## 已确认的计算细节

- 今天尚未打卡时，“当前连续”从昨天向前计算。
- 今天与昨天均未打卡时，“当前连续”为 `0`。
- 任务记录最多滑动到当前月，不能进入未来月份。
- 用户修改设备日期或时区后，按照新的设备本地日期重新计算。
- 预制任务从云端保存的统一生效日期起参与统计，换手机不得重置。

## 前端自测重点

- 同一天重复勾选和取消。
- 周六切换到周日时整周正确更新。
- 月初、月末、闰年二月和需要 6 行的月份。
- 今天同时存在背景色和黑色描边。
- `320px–480px` 连续响应式宽度、刘海屏和底部安全区。
- 关闭并重新打开 App 后数据保持。

## 前端结论

主要风险在日期算法、云端读写、PWA 缓存更新和嵌套滑动。固定用户名方案不提供真正的数据访问安全，这是 V1 已接受的限制。

## 本轮技术评审问题

- 保存流程已确认采用保存中状态，云端成功后更新界面，失败保持原状态。
- 日期箭头和任务颜色标识已确认按高保真稿实现：浅灰圆角日期按钮、任务颜色竖向圆角色块。
- 四个任务的三个月窗口已确认独立切换，单个任务滑动不影响其他任务。
- 百分比已确认四舍五入显示为整数，业务判断继续使用未取整原值。
