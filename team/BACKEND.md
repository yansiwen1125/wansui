# 万岁 V1.0 数据与后端需求对齐

## 角色

**后端工程师：云舟**

V1.0 使用 Supabase Postgres，不启用 Supabase Auth。该岗位负责单用户云端数据结构、同步冲突和统计规则。

## 已对齐需求

- 正式数据保存在云端，本地只保存只读缓存。
- 4 个固定任务由前端静态配置提供，不写入数据库。
- 同一任务同一天最多一条有效打卡记录。
- 在其他设备输入 `ysw` 可以恢复同一套记录。
- V1 只有一套数据，不提供多用户数据隔离。

## 推荐数据结构

### 系统任务

系统任务不建表，前端使用稳定且永不变化的固定 ID：

建议固定 ID：

- `preset_bowel`
- `preset_sleep`
- `preset_exercise`
- `preset_healthy_diet`

### 打卡记录

| 字段 | 说明 |
| --- | --- |
| `taskId` | 系统任务 ID |
| `date` | 设备本地日期，格式 `YYYY-MM-DD` |
| `completed` | 是否完成 |
| `updatedAt` | 数据库生成的最后同步时间 |

以 `taskId + date` 作为唯一键，重复操作只能更新同一条记录。

### 单用户设置

| 字段 | 说明 |
| --- | --- |
| `username` | 固定为 `ysw` |
| `schemaVersion` | 数据版本 |
| `firstInitializedAt` | 云端首次初始化时间，只写入一次 |
| `effectiveStartDate` | 固定为 `2026-06-10` |
| `lastKnownTimeZone` | 最近一次设备时区 |

`username` 设置唯一约束。初始化时使用原子写入：已存在则读取，不存在才创建。

## 已确认规则

### 1. 历史补打卡

- 用户可以补打卡 `2026-06-10` 至今天之间的日期。
- `2026-06-10` 之前不可新增、取消或修改打卡。
- 4 个预制任务从云端保存的统一 `effectiveStartDate` 起参与统计。
- 新设备登录时读取已有设置，不得覆盖首次初始化日期。

### 2. 统计分母

`总完成率 = 已完成次数 ÷（有效任务数 × 有效自然日数）`

本月有效日期从“本月 1 日”和 `effectiveStartDate` 中较晚的一天开始，到今天结束，不包含未来日期。

### 3. 当前连续的截止规则

- 今天已打卡：从今天向前连续计算。
- 今天未打卡：从昨天向前计算。
- 今天和昨天都未打卡：当前连续为 `0`，不查找更早的连续段。

### 4. 设备日期与时区

- 设备日期或时区变化后，按新的设备本地日期重新计算。
- `lastKnownTimeZone` 只用于排查和未来迁移，不用于锁定旧时区。

### 5. 入口与同步

- 用户名固定为 `ysw`，只作为前端入口判断。
- 不使用 Supabase Auth，不具备真正的身份验证和用户隔离。
- `taskId + date` 设置唯一约束。
- 同一记录发生多设备冲突时，以最后成功写入云端的操作为准。
- `updatedAt` 由数据库生成，不接受客户端自行传入。
- service role key 只允许存在于安全的服务端环境，不进入网页代码。
- 数据表只能授予 V1 所需的最小读写权限，不开放建表、删表或其他管理能力。

## 改进建议

- 统计逻辑集中在独立模块，不让不同页面各算一套。
- 所有日期先按设备本地日期归一化，再进行周、月和连续天数计算。
- 检测设备时区变化，但 V1 无需做复杂修复；按照当前时区重新展示即可。
- 初始化单用户设置时使用“存在则读取，不存在才创建”的方式，避免换设备后重置统计起点。
- `firstInitializedAt` 和记录 `updatedAt` 均由数据库生成，不接受客户端覆盖。
- 云端表和本地缓存都保留 `schemaVersion`，避免升级后无法读取旧数据。
- 为闰年、跨年、跨月和连续天数编写自动化测试。

## 数据结论

Supabase 承担数据库和数据接口，不单独开发自建服务器。固定用户名方案简单，但无法阻止知道接口的人读取或修改数据，这是 V1 明确接受的风险。

## 本轮技术评审问题

- 取消打卡已确认保留记录并保存 `completed=false`，用于支持最后写入生效。
- `effectiveStartDate` 已确认固定为 `2026-06-10`，不再根据设备或初始化时间计算。
- 无 Auth 时无法实现真正访问控制，只能限制表、字段和固定任务 ID。
- 已确认 V1 接受该安全风险，不增加密码、验证码或 Supabase Auth。
- 初始化必须走单事务 RPC，避免多设备同时首次打开产生不同生效日期。
