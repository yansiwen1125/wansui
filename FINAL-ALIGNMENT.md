# 万岁 V1.0 最终需求对齐

## 1. 会议结论

产品、设计、前端、后端和测试已完成最终联合评审。

- PRD、UI 和技术方案可以对应。
- 10 项技术问题均已确认。
- 当前没有阻塞开发的问题。
- V1.0 功能范围冻结，可以进入开发。

需求基准：

1. [产品需求文档](./PRD.md)
2. [高保真 UI](./design/high-fidelity/)
3. [UI 设计规范](./design/high-fidelity/UI-DESIGN.md)
4. [技术方案](./TECHNICAL-SOLUTION.md)

发生理解冲突时，先以 PRD 的功能规则为准；视觉细节以已确认高保真 UI 和 UI 设计规范为准；数据与工程实现以技术方案为准。

## 2. 产品经理确认

- V1.0 是 iPhone Safari 优先的 PWA，通过 GitHub Pages 发布。
- 只有固定用户名 `ysw`，手机记住登录状态。
- V1 接受无正式身份认证的安全风险。
- 只有 4 个固定任务，不提供任务管理、提醒、备注、社交或成就。
- 可操作日期固定为 `2026-06-10` 起（含当天）至今天。
- 月历为纯查看页面。
- 当前需求范围冻结。

产品结论：**通过，可以开发。**

## 3. 设计确认

- 四个任务名称、顺序和颜色固定。
- 首页、月历、任务记录及登录页高保真稿已确认。
- 日期箭头使用浅灰色圆角按钮背景，不显示描边。
- 任务卡片使用竖向圆角色块标识任务颜色。
- 月历使用极浅棕色表达完成率达到 50%。
- 月历与任务 Tab、底部导航、点击区域和安全区规则已明确。
- 加载中、离线只读、保存失败和用户名错误状态按 UI 规范实现。

设计结论：**通过，可以开发；开发完成后按高保真稿验收。**

## 4. 前端确认

- 使用 React、TypeScript、Vite 和 PWA。
- 首页、月历、任务记录共用同一份打卡数据。
- 日期统一使用设备本地日历日期，不以 UTC 截取日期。
- 打卡时先显示保存中，云端成功后更新界面；失败保持原状态。
- 每个任务独立维护三个月滑动窗口。
- 离线只读，不建立离线写入队列。
- IndexedDB 保存云端数据缓存，Service Worker 缓存应用外壳。
- 页面采用自适应宽度，适配 `320px–480px` 手机视口、iPhone 安全区和添加到主屏幕后运行。

前端结论：**通过，无技术阻塞。**

## 5. 后端确认

- 使用 Supabase Postgres，不建设自有服务器，不启用 Supabase Auth。
- 固定任务保存在前端配置，不建立任务表。
- `app_config` 保存唯一单用户配置。
- `effective_start_date` 固定为 `2026-06-10`。
- `checkins` 使用 `(task_id, date)` 唯一约束。
- 取消打卡保留记录并写入 `completed=false`。
- `updated_at` 由数据库生成，最后成功写入生效。
- 前端只使用 publishable key，不包含 service role key。

后端结论：**通过；安全限制已由产品接受。**

## 6. 测试确认

- 日期边界、自然周、月历、闰年、跨月和跨年必须自动化验证。
- 百分比四舍五入显示，50% 阈值使用未取整原值判断。
- 今天未打卡时从昨天计算当前连续；昨天也未打卡时为 `0`。
- 月历点击不跳转、不选中、不修改状态。
- 重点验证重复写入、取消打卡、缓存恢复、弱网失败和多设备最后写入。
- 必须进行 iPhone Safari、主屏幕 PWA、`320px–480px` 响应式宽度和安全区验收。
- 日期和统计严重问题清零后才能发布。

测试结论：**通过，验收口径完整。**

## 7. 已冻结规则

| 项目 | 最终结论 |
| --- | --- |
| 保存反馈 | 保存中，成功后更新 |
| 日期按钮 | 浅灰圆角背景，无描边 |
| 任务色标 | 竖向圆角色块 |
| 任务月份 | 每个任务独立滑动 |
| 取消打卡 | `completed=false` |
| 生效日期 | `2026-06-10`，包含当天 |
| 百分比 | 四舍五入显示整数 |
| 50% 判断 | 使用未取整原值 |
| 当前连续 | 今天未完成看昨天；昨天未完成则为 0 |
| 月历点击 | 纯查看，无动作 |
| 身份认证 | V1 不启用，接受风险 |

## 8. 开工条件

开发按以下顺序开始：

1. 工程、Supabase 数据结构、日期和统计模块。
2. 登录、首页打卡和云端保存最小闭环。
3. 月历和任务记录。
4. 缓存、离线与异常状态。
5. 自动化测试、iPhone 真机验收和 GitHub Pages 发布。

任何新增功能、规则变化或视觉结构调整均视为需求变更，需要先更新 PRD，再进入开发。
